# Demo 3 — `research_agent.py` (Multi-step Research Agent)

## What it does

A multi-step research agent that runs a **tool loop**: for a given topic it issues
several `web_search` tool calls (one per "aspect"), collects the snippets, then asks
OpenAI `gpt-4o-mini` to write a short briefing.

Flow per task:
1. `run_research()` (WORKFLOW span) kicks off a topic at a given `depth`.
2. `research()` (AGENT span) loops `depth` times, calling `search_with_retry()` each
   iteration to gather findings, then calls the LLM to synthesize a briefing.
3. `web_search()` (TOOL span) returns canned snippets — but sometimes fails first.

It runs 3 `TASKS` of increasing depth to trigger different neatlogs detections.

### Embedded signals
- **High tool count** — a deep task (`depth=8`) fires many `web_search` calls
  → `tool_call_count` condition.
- **Tool error + retry** — `web_search()` raises `ConnectionError` on ~1 in 3 first
  attempts, then `search_with_retry()` retries with backoff and succeeds
  → **ERROR span** on the failed attempt → Execution Failed / fixes.
- **Expensive / token-heavy run** — the deep task accumulates many findings, driving
  up the summary prompt size → cost / token condition.
- **Latency** — each search sleeps a random `0.2–2.5s` → latency condition.

## Key behavior: error + retry

```python
@span(kind="TOOL", name="web_search", tool_name="search.web")
def web_search(query, attempt=1):
    ...
    # Fail on the first attempt for one in three searches, then succeed on retry.
    if attempt == 1 and random.random() < 0.33:
        raise ConnectionError(f"search backend timed out for: {query}")
    return random.sample(FAKE_RESULTS, k=3)


def search_with_retry(query):
    max_attempts = len(RETRY_BACKOFF_SECONDS) + 1
    for attempt in range(1, max_attempts + 1):
        try:
            return web_search(query, attempt=attempt)
        except ConnectionError as e:
            print(f"[retry] attempt {attempt} failed: {e}")
            if attempt >= max_attempts:
                raise
            time.sleep(RETRY_BACKOFF_SECONDS[attempt - 1])
```

The failed attempt propagates through the `@span`-decorated `web_search`, so that
span is marked **`status = ERROR`** — even though the retry ultimately succeeds. This
is what makes research_agent one of the reliable **error / "fixes"** demos (alongside
`flaky_api_agent.py`).

## Fix generated by the neatlogs dashboard

The **Neatlogs Agent** read the ERROR spans and auto-generated a Fix card:

> **Fix Intermittent web_search Tool Timeouts and Retry Latency**
> - Status: Open · Severity: high · Category: Tool · Pattern: Latency anomaly
> - Source: Neatlogs Agent · Workflow: research-agent
> - Confidence: 65% (medium — strong evidence, low pattern history)

**Root cause (per dashboard):** `web_search` hits intermittent `ConnectionError`
timeouts from the external backend; `research_task` recovers on attempt 2, but the
first failure spikes the error rate and latency.

**Evidence:** trace `3b5834d348afb19cace91d036c515ea4` — `web_search` fails on
`attempt: 1` (`ConnectionError: search backend timed out for: electric vehicle
battery trends — aspect 1`), succeeds on `attempt: 2`. Trace
`327ad2c4a250e9549a28e5601b630b79` shows a clean first attempt, confirming the
intermittent nature. Linked traces (2), both `research-agent` · Jun 5, 2026 · 10:07 PM.

**Recommended action — and how it maps to the code we then added:**

| Fix recommendation | Code added to `research_agent.py` |
|---|---|
| Increase the tool timeout to **30s** | `SEARCH_REQUEST_TIMEOUT = 30` |
| Use **exponential backoff (1s, 3s, 5s)** for retries | `RETRY_BACKOFF_SECONDS = [1, 3, 5]` |
| Verify backend health / **rate limits** | `SEARCH_RATE_LIMIT_PER_MINUTE = 10` + `_respect_search_rate_limit()` / `_check_search_backend_health()` |
